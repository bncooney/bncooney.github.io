Claude Code Opinionated Tips and Tricks

March 20, 2026

Claude Code (the application) with Claude Opus (the model) is easily the best-in-class coding assistant available today. After spending real time with it, here are the tips and tricks that have made the biggest difference in my workflows.

1. Use the VS Code extension, avoid the terminal application

Claude Code started as a CLI tool, and many tutorials still present it that way, but the VS Code extension is an enormously better experience for day-to-day work.

What you get in VS Code:

  • Reliability. The terminal app uses a React-based renderer with notoriously poor performance. The document model was built for rendering anywhere else. The VS Code extension runs in VS Code's native V8-backed webview, which handles large outputs and extended conversations without issue.
  • Interoperability. Claude's output integrates with your existing extensions:
    • Rendered Markdown previews
    • PlantUML diagrams
    • Existing filetype hooks and language servers
  • Consistency. Split your editor so Claude's conversation sits beside the files it's changing. Review diffs inline, comment on plans in context, and keep your file explorer and source control panel open while Claude works.

The terminal is great for scripting, CI pipelines, and piping output into Claude. But for interactive coding sessions, VS Code removes friction at every step. Install the extension, open the command palette, and search "Claude Code" to get started.

2. Skip /init. Let memory do the work

Many guides tell you to run /init and set up a CLAUDE.md (or copilot-instructions.md, etc.).

Don't.

These files get stale and require ongoing management, and every update invalidates the context cache. This hurts you in two ways:

  • Anthropic's token cache (the cheapest tokens you have) gets busted each time the file changes, so frequent edits mean frequent cache misses.
  • Stale or incorrect instructions poison your conversation context.

Instead, lean on Claude Code's built-in memory system. It lives in ~/.claude/projects/.

  • Don't state the obvious. Claude Opus is a monster. Instructions should address consistent incorrect behaviours that you've observed over multiple conversations, but keep in mind a lot of "mistakes" Claude makes stem from trying to follow established patterns in your codebase. (More on this later.)
  • Keep this short, sharp, and to the point. Only 200 lines will be loaded into context.
  • As a result, updates should be infrequent, and considered.

The MEMORY.md index is always loaded into context, so Claude always has this available. This includes sub-agents, without a single config file checked into your repo. You can explicitly say "remember this" to save something.

This approach keeps your repo clean, avoids conflicts with teammates who use different tools, and produces context that's actually personalized to how you work.

3. Vibes only. Let deterministic tools do deterministic work.

Don't ask Claude to lint your code. Don't ask it to fix formatting. Don't put lint rules in your memory files. These are deterministic problems with deterministic solutions. If your toolchain doesn't handle them, fix your toolchain.

Every token Claude spends reasoning about whitespace, semicolons, or import ordering is a token not spent on the thing it's actually good at: understanding intent, designing solutions, and writing code. Linting instructions in context are doubly wasteful. They burn the prompt, and they burn the output when Claude "fixes" things your formatter would have caught on save.

Set up your deterministic tools and let them run:

  • Formatters (Prettier, Black, dotnet format). Run on save or as a pre-commit hook.
  • Linters (ESLint, Ruff, Roslyn analyzers). Run in CI or as a background task in your editor.
  • Type checkers (TypeScript, mypy, the C# compiler). These catch entire categories of bugs before Claude ever needs to think about them.

Claude's job is authoring. Intent, architecture, code: the stuff that can't be reduced to a rule set. Keep it focused there. If your formatter disagrees with Claude's output, the formatter wins automatically. That's the whole point.

4. Tech debt just got more expensive

Claude learns from your codebase. Its superpower is reading.

When Claude reads your project, it treats existing code as the authority on how things should be done. If your codebase has a leaky abstraction, Claude will leak the same way. If you've got a service class that mixes data access with business logic, Claude will write the next one exactly like it. It's not making a mistake. It's following your lead.

This means tech debt compounds differently now. Before, a bad pattern was a maintenance burden for your team. Today, it's a template that gets replicated at Shenzhen speed. Every anti-pattern in your codebase is an anti-pattern Claude will faithfully reproduce, across every new file it touches.

The practical implications:

  • Fix the pattern, not the output. If Claude keeps producing code you don't like, check whether it's copying something that already exists in your project. Correcting Claude in conversation is a band-aid. Maintenance is the fix.
  • Prioritise refactoring differently. That "we'll clean it up later" code now actively degrades every AI-assisted change. The ROI on paying down tech debt just went way up.
  • Use Claude to do the refactoring. This is the silver lining:
    • Point Claude at the bad pattern.
    • Explain what the good pattern looks like.
    • Let it do the tedious migration across your codebase. It doesn't get bored or distracted and start making mistakes.
    Once the codebase reflects the right approach, Claude will follow it going forward.
  • Point Claude at the good examples, hide the bad ones. If your codebase has both good and bad patterns, don't leave it to chance which one Claude latches onto.
    • Explicitly reference the files that do it right.
    • Consider adding extremely poor examples to the Read deny list in your settings.
    • If Claude can't see the bad examples, it can't copy them.

The quality of Claude's output is a mirror of your codebase's quality. Good code in, good code out.

5. Plan and execute in the same conversation, and don't use Plan mode

The traditional workflow is: plan in one context (conversation), execute in another. Important context gets lost in the handoff every time.

With a 1 million token context on Opus, planning and execution should happen in the same conversation.

Counterintuitively, don't use Plan mode for this. When you toggle between Plan and Execute, Claude's system prompt changes. That prompt change invalidates the context cache, which means Claude re-reads and re-explores the codebase from scratch when you switch modes. You pay for the same understanding twice.

Instead, just leave Claude in normal mode with "Ask before Edits" enabled. Describe your task, let Claude explore and propose an approach, and discuss it. Iterate, refine, ask questions. Claude will execute with the full conversation history intact, diffs will be presented for your approval before they're applied, and it never had to re-orient itself.

This changes things:

  • Decisions carry forward. When you discuss trade-offs during planning, Claude carries that reasoning into the implementation. No re-explaining, no drift.
  • Corrections compound. If you steer Claude away from an approach during the plan ("don't introduce a new dependency for this"), it won't circle back to that approach when writing the code.
  • The 1M token context makes this practical. Claude Opus gives you a million tokens of context. That's enough room to plan a substantial feature, discuss the edge cases, and then implement it, all without losing the thread. And the price is the same regardless of context length.

The workflow looks like this:

  1. Describe what you want to build.
  2. Ask Claude to plan the approach. Iterate.
  3. Once you're happy, tell it to execute. Every decision you made in steps 1 and 2 is still in context.

This is faster because Claude doesn't need to read your codebase twice. In a split workflow, the planning conversation reads the relevant files, builds up understanding, and then that context is thrown away. The execution conversation starts cold, re-reads the same files, and has to reconstruct the same understanding from scratch. That's redundant work you're paying for in time and tokens.

It also produces better code, because the implementation is informed by a conversation that the executor was present for. There's no lossy translation between a plan document and the person (or agent) writing the code. The executor is the planner.

6. Clone your dependencies

Claude Code can fetch code from GitHub etc., but it's slow, unreliable, and rate limited.

There's a much better option for open source dependencies: just clone them. Remember: Claude's superpower is reading.

With the repo locally, Claude can:

  • Grep the actual source code, not a fallible summary of it.
  • Read type definitions, interfaces, and signatures directly.
  • Follow the implementation through to understand behaviour, not just the documented API surface.
  • Search for usage examples in the project's own test suite.

While Claude Code has a working directory it doesn't mind references from other locations, so you can point it at your project and your dependency's repo simultaneously. When Claude needs to understand how a library works, it reads the source. No network round-trips, no stale docs, no tool parsing errors.

This is especially valuable for:

  • Libraries with poor or outdated documentation.
  • Internal forks or patched versions where the public docs don't match.
  • Understanding edge cases that only the source code reveals.

A git clone takes seconds. The accuracy improvement lasts the entire session.

7. Fork your conversations aggressively

Maintain the context you've built up by forking the conversation at an earlier point. This is particularly useful for solving multiple issues as a result of a single change, without the risk of compaction.

Forking is time and dollar cost efficient, because once again, you're maintaining the same cache.

The instinct from chat-based tools is to keep one long conversation going. Fight that instinct. Conversations are free. Tokens are not.

8. Never compact.

At the time of writing, compaction is strictly useless. It summarises your conversation history using a smaller, less capable model, then replaces the actual history with that summary. You lose nuance, you lose corrections, you lose the reasoning that informed decisions. Everything you built up in tips 5 and 7 gets flattened into a lossy paragraph.

If your conversation is getting long, you have two better options:

  • Fork at an earlier point. Go back to before the conversation went off track and branch from there. You keep the good context and lose the noise.
  • Start fresh. Give Claude the context it actually needs: a summary you wrote yourself, the current state of the plan, or just a git diff. You know what matters better than a compaction model does.

Compaction feels productive because it keeps the conversation alive. But a conversation full of summarised context is worse than a new conversation with precise context. Don't let sunk cost keep you in a degraded session.

9. Commit often, squash later

Claude works best when it can see what's changed. Committing frequently gives both you and Claude clean checkpoints to reason against.

Build an inherently resilient workflow:

  • Safe rollback points. If Claude goes in a wrong direction, you can reset to the last good commit instead of trying to untangle which changes to keep.
  • Better diffs for new conversations. When you start a fresh conversation (per tips 7 and 8), a git diff against your last commit gives Claude a precise summary of where things stand. No explanation needed.
  • Cleaner reviews. Small, frequent commits during development don't mean small, frequent commits in your PR. Squash them before merging. The development history served its purpose; the review history should be clean.

Think of commits during a Claude session like quicksaves in a game. Save often, save before anything risky, and don't worry about the save file names until you're done.

Wrapping Up

  1. Use VS Code. It's more reliable, more interoperable, and easier to manage than the terminal.
  2. Skip /init. Let memory build up organically instead of maintaining stale config files.
  3. Vibes only. Let your toolchain handle linting, formatting, and type checking. Keep Claude focused on authoring.
  4. Pay down tech debt. Claude follows your codebase's lead. Anti-patterns now replicate at machine speed.
  5. Plan and execute together. Same conversation, same context. Don't use Plan mode.
  6. Clone your dependencies. Local source beats web fetching every time.
  7. Fork aggressively. Branch your conversations like you branch your code.
  8. Never compact. Fork earlier or start fresh with precise context instead.
  9. Commit often, squash later. Frequent commits give you rollback points and clean diffs for new conversations.

Bonus: All of this will change

Every tip in this post has a shelf life. Boris Cherny and the Claude Code team ship at a pace that makes most of these observations temporary. Plan mode might get context-aware caching tomorrow. Compaction might become lossless next month. The VS Code extension might get features that make half of this list redundant.

The only durable advice is: stay adaptive. Hold your workflows loosely. When the tooling changes, change with it. The developers who get the most out of Claude Code aren't the ones who found the perfect setup. They're the ones who keep adjusting.

Be like bamboo. Strong roots, flexible everything else.
