Everything, all over the place, constantly

I had meant to describe the tech behind the blog, and iterate on the static site generator, but the effort was largely undirected, and undirected effort is often wasted. So I started iterating on one of my second order motivating factors in writing a "tech" blog, which is that it would be really fun to make a little game to prove to my nephew that I'm cool.

I've used Unreal for another personal project I'd love to talk about on another day, and I'd used Unity in a professional context, but for the sake of my own personal interest (and in "finishing" something), its high time I made a more simple (and incredibly over-engineered when the whim takes me) pixel art game.

For a myriad reasons, I've decided to use MonoGame. I'm sure I'll touch on why in more detail at some point, offline or otherwise, but the tldr is that it provides (more than) just enough to get the paperwork out of the way (an asset pipeline alone would be more than enough complexity, that is not particularly interesting) while still being entirely hands off. Its also all code, all the time, and C# at that, so it interacts well with code generation of all flavors (source generators, AI etc.).

My first thought was to build a tileset and map editor in MonoGame as well, and I still might (I have some code loading in EF Core to use SQLite to store game text data, asset metadata, configuration etc. before its "built"), but to get moving I've been playing with Tiled, and its great! As a systems-ey programmer, obviously the first thing I did in earnest was write the XML / Json deserialiser for importing that into the game project, and in doing so I already got some learning value from the exercise re: .NET AOT, which is really nice as well.

Dog food all the way down

In poking around with game project, and having something I wanted to share, I exposed another feature I want / need in the static site generator. Images! Right now, I can throw images at the output directory, but thats an awful workflow, and that is to say nothing of their suitability for the web (i.e. they could be LARGE). So I haven't tonight, but I need to add something to push these in one end of the build chain and have them come out the other. This is very simple, and the challenge is in doing this intelligently, and getting results that are actaully an improvement. The image below for example, is a few hundred kilobytes as a .png, and flawless, but my expectations that I could just throw some jpeg compression at it and call it a day were false. It isnt meaningfully smaller (and your browser is probably capable of something like gzip compression as well anyway), and the quality of the image afterwards was noticeably poorer.

All of which is to say, this is exactly why I was glad for the build pipeline in MonoGame, and didn't want to roll my own.

Everything is related!
