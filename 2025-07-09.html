On writing a static site generator in the year of our Lord twenty twenty five

The first webpage I ever made was written in Microsoft Word and was about the Ghostbusters. I didn't get very far, and I don't recall if it ever made it to GeoCities (as God intended). If it did, I don't expect the internet is any worse of for its loss to the passage of time.

That isn't to say that the internet is better off without GeoCities. On the contrary, the internet has become much less personal, but much more invasive. Many more people have access, but it has never become any easier than it was in the late nineties to create a webpage.

I would digress here into a (very casual) examination of the exhaustive monetisation of ...everything, the ephemeral nature of short form video, and the causal loop of social media on the aformentioned, but I would never finish the post.

Static Site Generation

My design intent for writing a static site generator:

  • I want content to load fast.
    • Computers are fast now. FAST, fast.
    • All this waiting around is a software problem.
  • I want to be able to write fast.
    • I want to write words on a page, with paragraphs and inline images, without chasing layout around like I did in 1997.
    • Don't cross the streams. WYSIWYG is a great idea, but layout and rendering are not the same thing.
    • I should be able to render everything side by side, and updates should be at interactive speed (see above).
  • I want freedom.
    • I want to be able to do anything, anywhere, for any reason... browser permitting.
    • WordPress etc. are cool, and there is already an extension for everything, but nothing I want to do is so complex it can't be implemented in the time it would take to find an extension and learn a (terrible) new API
    • Also, see; "monetisation of everything"

Writing a static site generator, with the constraint that I was writing it for non-general use (i.e. for myself), is entirely straightforward; load HTML parts and assemble (its practically php with fewer steps). CSS is still the devil, but the devil is in the toaster now, and code AI is perfect for stochastic generation when the outcome is practically stochastic itself (I jest, but CSS is no fun).

Site generation is implemented in C#. In one big soup of pure creation, dotnet watch builds on save and assembles the output, which is directed to the local git repo. Live Server watches that directory and renders the changes locally so I can see the mess I'm making, and when I'm happy with the changes I push to remote and GitHub actions builds the page.

AI generation handles the CSS almost exclusively. The value in generating everything, all at once, is that it can get it wrong as many times as it takes to get it right, it updates in (almost) real-time through Live Server, and I can unplug the toaster before it overcooks anything.
